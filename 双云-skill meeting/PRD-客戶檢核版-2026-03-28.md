# Skill Meeting PRD
版本：Client Review Draft
日期：2026-03-28
狀態：依目前專案實作重新整理

## 1. 文件目的
本文件用於對齊 Skill Meeting 目前的產品定位、功能範圍、實作進度與下一階段規劃，供內部檢核後提供給客戶。

本版 PRD 不再沿用舊版的「理想規劃」描述，而是以目前專案資料夾內可辨識的主線版本與實際程式結構為基礎整理。

## 2. 產品定位
Skill Meeting 是一個讓多位 AI 角色針對同一議題進行多輪圓桌討論的應用。使用者可以建立或匯入不同專業角色，指定討論主題與背景資料，啟動多輪討論後，自動生成結構化報告，作為企劃討論、商業分析、策略發想、教學設計與顧問提案的輔助工具。

## 3. 核心價值
1. 讓單一提問轉化為多角色、多視角的結構化討論。
2. 讓使用者可以自訂 AI 角色，而不是只使用單一固定助理。
3. 讓討論結果自動整理成可保存、可下載、可複製的報告。
4. 延伸為可分享 Skill 的市集型產品，建立角色資產與方法論資產。

## 4. 目標使用者
1. 顧問與企劃人員：需要快速收斂多方觀點與策略方向。
2. 企業內部團隊：需要模擬跨部門會議或方案審查。
3. 教學與培訓場景：需要用多角色方式進行案例討論。
4. 進階 AI 使用者：希望建立自己的專家角色庫並重複使用。

## 5. 主要使用情境
1. 輸入一個商業或專案問題，讓多位 AI 角色進行 2-5 輪討論。
2. 匯入自定義 Skill，模擬特定顧問、專家或方法論角色。
3. 針對單一主題生成會議報告，作為後續簡報或提案基礎。
4. 從 Marketplace 安裝其他人分享的 Skill，擴充討論角色庫。
5. 使用 SOSTAC 類型的預設角色群，進行行銷策略分析。

## 6. 產品範圍
### 6.1 目前主線產品
目前資料夾中可辨識的主線版本為 `skill-meeting/`，其特徵如下：

1. 採用 Next.js 16 + React 19 + TypeScript。
2. 使用 App Router。
3. 會議資料、使用者自訂 Skill、Marketplace 發佈內容主要存於瀏覽器 localStorage。
4. AI 討論透過 Next.js API Route 呼叫 Anthropic SDK 並以 SSE 方式回傳前端。
5. 已有 Marketplace 前台與發佈、安裝流程，但仍屬本地原型階段。

### 6.2 非主線資料夾判讀
根目錄中同時存在多組資料，推測為不同階段的演進與備份：

1. `skill-meeting/`
目前最完整、最接近對外展示的主線版本。

2. `skill-meeting-clean/`、`程式碼/`
較早期的前端版本，結構相近，較像整理中的舊版或備份。

3. 根目錄 `server/`、`src/`、`ARCHITECTURE.md`
屬於另一條規劃中的架構方向，描述 React + Express + Prisma + SQLite 的正式化版本，但與 `skill-meeting/` 現況不一致，應視為下一階段架構草案，而非已完成狀態。

4. `sostac-integration/`
屬於 SOSTAC 角色擴充與整合過程的中間資料。

## 7. 功能需求
### 7.1 首頁與會議列表
使用者進入首頁後，可看到既有會議列表；若尚未建立會議，系統會顯示空狀態並引導建立新會議。

需求重點：
1. 顯示主題、建立日期、參與角色數、討論輪數與狀態。
2. 支援從首頁快速進入新會議建立流程。
3. 已完成的會議可導向報告頁。

### 7.2 建立新會議
使用者可建立一場新的 AI 圓桌討論。

需求重點：
1. 輸入討論主題。
2. 輸入背景資料或來源資料。
3. 選擇討論目標：達成共識、探索問題、腦力激盪。
4. 選擇討論輪數。
5. 選擇 2-6 位參與 Skill。
6. 建立後導向會議室。

### 7.3 Skill 管理
Skill 是此產品的核心資產。使用者可檢視、建立、匯入、刪除與啟用/停用 Skill。

需求重點：
1. 顯示預設 Skill 與自訂 Skill。
2. 預設 Skill 可啟用/停用。
3. 自訂 Skill 可新增、刪除。
4. 支援 JSON 匯入。
5. 支援搜尋與篩選。
6. 支援從 Skill 管理頁進一步發佈到 Marketplace。

### 7.4 討論引擎
討論引擎是產品核心體驗。系統會將會議設定與角色設定送入 AI，讓角色依序輸出內容。

目前設計流程：
1. Opening：每位 Skill 先提出初始觀點。
2. Discussion：依設定輪數進行多輪回應。
3. Summary：由系統角色彙整報告。

需求重點：
1. 保持每個角色的人設、專長與風格。
2. 讀取主題、來源資料與前文脈絡。
3. 逐段串流顯示內容。
4. 討論完成後生成報告資料結構。

### 7.5 會議室
會議室負責呈現即時討論過程與進度。

需求重點：
1. 顯示會議標題與當前階段。
2. 顯示參與角色頭像與當前發言者。
3. 即時顯示訊息串流。
4. 顯示進度條。
5. 討論完成後同頁展示報告。

### 7.6 報告輸出
系統在討論結束後自動產出結構化報告，並提供 Markdown 下載。

目前報告結構：
1. 會議資訊
2. 共識結論
3. 分歧觀點
4. 待釐清問題
5. 完整逐段對話
6. 各角色簽署

### 7.7 Marketplace
Marketplace 是產品第二條增長線，讓使用者可以發現、安裝、發佈 Skill。

目前需求重點：
1. 市集首頁可顯示精選、熱門下載、最新發佈。
2. 提供搜尋與分類。
3. 可查看 Skill 詳情。
4. 可將 Marketplace Skill 安裝到本地 Skill 庫。
5. 可發佈新的 Marketplace Skill。

## 8. 當前實作進度
### 8.1 已完成或已有可操作原型
1. 首頁與會議列表。
2. 新會議建立頁。
3. Skill 管理頁。
4. 預設 Skill 與自訂 Skill 的 localStorage 儲存。
5. 討論 API 與前端 SSE 接收流程。
6. 會議室即時訊息顯示。
7. 討論完成後的報告顯示與 Markdown 下載。
8. Marketplace 首頁、搜尋頁、詳情頁、發佈頁。
9. Marketplace 本地安裝流程。
10. SOSTAC 系列預設 Skill 已加入主線資料。

### 8.2 已有雛形但仍屬原型或假資料
1. Marketplace 主要依靠 seed JSON 與 localStorage，尚非正式後端資料源。
2. Marketplace API Route 已存在，但前台多數讀取仍直接走 client-side store。
3. 發佈與安裝流程已通，但沒有帳號、審核、版本治理與權限控管。
4. 下載量、評分、作者資訊目前為展示資料結構，不是正式交易或社群系統。

### 8.3 尚未完成
1. 使用者登入與權限系統。
2. 正式資料庫持久化。
3. 多人協作與分享。
4. 付費與金流。
5. 評論與評分機制。
6. 後台管理。
7. 檔案型來源資料上傳。
8. 正式部署後的用量控管與安全治理。

## 9. 真實技術現況
### 9.1 目前實際技術架構
1. 前端：Next.js App Router。
2. 伺服端邏輯：Next.js API Routes。
3. AI 服務：Anthropic SDK。
4. 即時回傳：SSE 格式串流。
5. 本地儲存：localStorage。
6. 主資料來源：`default-skills.json` 與 `marketplace-seed.json`。

### 9.2 與舊版文件的差異
資料夾內另有一份較完整的正式架構規劃，內容包含 Express、Prisma、SQLite、JWT 與 Admin 後台；但那是規劃中的下一階段，不是目前 `skill-meeting/` 主線已完成內容。對客戶溝通時，應避免把該架構草案當成已交付功能。

## 10. 建議對客戶描述的產品階段
目前較適合定位為：

「可展示、可操作的 MVP / Prototype Plus」

原因：
1. 主流程可跑通。
2. 有明確產品體驗與差異化。
3. 已有第二模組 Marketplace。
4. 但資料層、帳號層、正式治理層仍未完成。

## 11. 下一階段需求建議
### Phase 1：產品可商用基礎化
1. 導入帳號系統。
2. 將會議、Skill、Marketplace 發佈資料從 localStorage 轉入資料庫。
3. 完成正式後端資料存取層。
4. 補齊錯誤處理、驗證與權限。

### Phase 2：Marketplace 正式化
1. 建立作者身份與個人頁。
2. 支援 Skill 版本管理。
3. 加入評價與下載統計。
4. 建立審核與上架機制。

### Phase 3：商業化
1. 付費 Skill。
2. 企業版角色庫。
3. 團隊共享會議與報告。
4. 客戶專屬模板與白牌化輸出。

## 12. 風險與限制
1. 目前資料存於瀏覽器，換裝置或清除快取即可能遺失。
2. 目前沒有登入與權限隔離，不適合正式多人共享。
3. Marketplace 目前偏展示與原型驗證，尚未形成正式平台機制。
4. 若要商用，需補強資料治理、用量管理、錯誤監控與安全性。
5. 目前部分舊文件與部分介面文字存在編碼混亂情況，正式對客文件與 UI 文案需再統一整理。

## 13. 驗收建議
建議用以下角度檢核本產品是否符合客戶期待：

1. 核心價值是否清楚：多角色 AI 討論是否真的比單一聊天更有價值。
2. 主流程是否流暢：建會議、選角色、討論、看報告是否一氣呵成。
3. Skill 是否具有可擴展性：是否能支援客戶自己的角色與方法論。
4. Marketplace 是否值得成為第二階段主軸。
5. 客戶需要的是展示型 MVP、內部使用工具，還是正式 SaaS，三者預算與時程差異很大。
